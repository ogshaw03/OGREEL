## ⚠️ 未解決課題: Firestore データの蓄積（2026-06-06 記録）

### 課題
ローカルフォルダ連携は「**共有データ(Firestore)は追加のみで削除しない**」方針のため、
ローカルのファイル/フォルダを削除しても **Firestore 上の動画・フォルダのドキュメントは残り続ける**
（各自の画面では存在ベースで非表示にしているだけ）。

このため、長期運用で **不要ドキュメントが蓄積** し、

- ドキュメント数の増加によるクエリ/読み取りコストの増大
- ストレージ・サーバー負荷への悪影響

が将来的に問題になる可能性がある。

### 現状の判断
- 今は **このまま進める**（削除すると他メンバーがまだ持つローカル動画/フォルダが共有から消えるリスクがあるため、安全側に倒している）。

### 今後の検討（要対応）
- 安全にガベージコレクションする仕組み（例）:
  - どのメンバーの実体にも存在しないローカル動画/フォルダを、一定条件で運営が一括削除できる管理機能
  - 「最後に存在を確認したメンバー/日時」をドキュメントに持たせ、全員が一定期間“不在”なら削除候補にする
  - 運営コンソールに「孤児ドキュメントの棚卸し/削除」ツールを追加
- 削除を行う場合は **per-user の存在情報を集約**してから判断し、誤削除（他メンバーがまだ保持）を防ぐこと。

> 関連: 下記「ローカルフォルダとアプリ表示の同期モデル」の大原則（追加のみ・view-only）。

## ローカルフォルダとアプリ表示の同期モデル（2026-06-06 記録）

ローカルフォルダ連携まわりは設計が独特で、不具合が再発しやすいので重要事項として記録する。

### 大原則（このモデルの肝）
- **共有データ(Firestore)は基本「追加のみ」で削除しない。** あるユーザーのリロード/同期で他メンバーがまだ持っているローカル動画・フォルダが共有リストから消えるのを防ぐため。
- **「実体に合わせた更新」は各自の“表示のみ”で行う（per-user / view-only）。** Firestore の文書は消さず、自分の画面だけをローカルフォルダの実体に追従させる。
- **フォルダはリアルタイム購読しない。** 入室/リロード時に `getDocs` で一度だけ取得（`loadFolders`）。自分の操作後のみ自分の画面を再取得。→ 他ユーザーの変更/リロードが即波及しない。
- **ローカル動画ファイルはユーザーごと**（File System Access のハンドルは転送不可）。各自が同じフォルダを接続する前提。

### 関係する主な関数
- `localFileExists(v)`: ローカル動画 `v` の実ファイルが接続フォルダ内に在るか。**非ローカル動画は常に true**。判定は `localFileHandles` / `localLowHandles`（スキャンで構築）。
- `getFilteredVideos()`: 表示用動画。**ローカル接続中(`currentDirHandle`)のみ** `localFileExists` で実体の無いローカル動画を除外。
- `countVideosUnder(path)` / `getFolderItemsAtLevel(path)`: フォルダの件数・一覧。**接続中は `localFileExists` 基準**で数え、存在動画が無いフォルダは一覧から除外（描画時フィルタ）。
- `loadFolders()`: フォルダを getDocs で一度だけ取得。`refreshLocalFolder` / `autoSyncLocal`: リロード時の再スキャン＆追加同期（削除はしない）。

### 過去に起きた不具合と原因（再発時のチェックポイント）
1. **「ローカルの中身を削除してもフォルダが“動画あり”と表示される」**
   - 原因: `countVideosUnder` が**生の `videos`(Firestoreドキュメント)を数えていた**。ファイルを消してもドキュメントは残る（追加のみ運用）ため件数が残った。タイルは `getFilteredVideos` で隠れるのにフォルダ件数だけ生データ基準、という不整合。
   - 対策: フォルダの件数・表示判定を**実ファイル存在基準（`localFileExists`、接続中のみ）に統一**。`getFolderItemsAtLevel` で存在動画ゼロのフォルダを描画時に除外。**配列を一度だけ書き換えるプルーニングはタイミング依存で不安定なので使わず、描画時フィルタに一本化**。
2. **「誰かのリロードで他ユーザーのフォルダ一覧が勝手に更新される」**
   - 原因: フォルダを `onSnapshot` でリアルタイム購読していたため、`autoSyncLocal` のフォルダ書き込みが全員に波及。
   - 対策: フォルダ購読を廃止し `loadFolders`（getDocs 一度きり）へ。自分の操作後のみ自分の画面を再取得。
3. **「フォルダ更新がグリッドに反映されない」**
   - 原因: `renderGrid._sig`（再描画スキップ署名）が動画フィールドのみで、フォルダ名/色/増減を見ていなかった。
   - 対策: 署名にフォルダ状態(`id/name/color/parent`)を含める。
4. **「リロード時にローカルフォルダ内のフォルダが表示されないことがある」**
   - 原因: `loadFolderHandle` がスキャン完了後に `renderGrid()` のみ呼び、サイドバーのフォルダ一覧(`renderSidebar`)を再描画していなかった。スキャン前(`localFileHandles`が空)に描画されると存在判定 false でフォルダが隠れ、その後再描画されず残る。
   - 対策: スキャン完了後を `renderAll()` に変更（フォルダ一覧も再描画）。
5. **「ローカル動画を削除→再追加すると2つに重複する」**
   - 原因: `dedupLocalVideos` が `new Set()` に対し `.set()`（正しくは `.add()`）を呼んでおり、最初のローカル動画で TypeError → 重複除去が一切機能していなかった。判定元も onSnapshot 未反映の古い `videos` で取りこぼしていた。
   - 対策: 最新状態を `getDocs` で取得し、`.add` で URL 重複を判定。`createdAt` 降順で同一URLは最新1件を残して古い重複を削除。`refreshLocalFolder`（リロード時）で毎回実行。

### 再発時の指針
- フォルダの「件数・表示・有無」は **必ず `localFileExists` を通した存在ベース（接続中）** で判定すること。生の `videos` を直接数えない。
- 「実体に合わせて消す」処理は **Firestore を消さず、表示フィルタ（描画時）で実現**すること。
- ローカルフォルダのスキャン後は **`renderAll()`** で必ずサイドバーも再描画すること（`renderGrid()` のみにしない）。
- 重複判定は **最新の getDocs 結果**で行い、コレクション型のメソッド（`Set.add` / `Map.set`）を取り違えないこと。
- フォルダはリアルタイム購読に戻さないこと（戻すと #2 が再発）。

## Firestore 永続キャッシュの挙動（2026-06-06 記録）

dev に導入した **オフライン永続キャッシュ（IndexedDB）** の挙動まとめ。

実装：
```js
const db = initializeFirestore(app, {
  localCache: persistentLocalCache({ tabManager: persistentMultipleTabManager() })
});
```

### キャッシュが「残る」ケース（再利用され、Reads を抑え続ける）
- PC再起動 / ブラウザ再起動 / タブを閉じる → そのまま保持
- 次回ロード時：**キャッシュから即表示 → サーバーには差分のみ取得**
- IndexedDB はディスク保存のため、セッションをまたいで永続する

### キャッシュが「消える」ケース（次回は初回ロード扱い＝全件1回読み直し）
- ブラウザで「サイトデータ／Cookie・キャッシュの削除」を実行
- シークレット／プライベートモード（セッション終了時に破棄）
- 別のブラウザ・別端末・別プロファイル（キャッシュはブラウザ×オリジン単位）
- ディスク容量逼迫時のブラウザによる自動退避（まれ）

### 消えても問題ない理由
- データの正本は常に Firestore 側にあるため、**キャッシュが消えても失われるデータはない**
- 消えた次の1回だけ全件読み込み（＝初回と同じ）→ 以降また自動でキャッシュ再構築
- 古いキャッシュが残っていても、接続時にサーバーと照合して最新へ更新される（古い内容が出続けることはない）

### 結論
通常運用ではキャッシュは生き続け、リロード起因の Reads を継続的に抑制する。
効果が一時的にリセットされるのは「サイトデータ削除」「シークレットモード」等の限られたケースのみ。
